**Status:** Complete. Working corpus is **1,713 films with all four
signals** (screenplay + budget + revenue + rating), comfortably in the
"≥ 1,500: proceed with MovieSum-only" green band of the corpus-size
decision criteria (`PROJECT_CONTEXT.txt` Section 5). Some end-of-phase
questions are still open and should be brought to the planning

**Date completed:** 2026-05-02

## What we did

1. Set up the project skeleton (`src/`, `data/`, `reports/`, `docs/`,
   `tests/`, `notebooks/`), `.gitignore` excluding raw data and standard
   Python artifacts, `src/utils/paths.py` (project-root resolver via
   `docs/PROJECT_CONTEXT.{txt,md}` markers) and `src/utils/logging.py`
   (single shared stdout handler, default level WARNING — set
   `MSBA315_LOG_LEVEL=INFO` to see operational messages).
2. Loaded the IMDb-TMDB Movie Metadata Big Dataset (1M)
   (`src/data/load_ratings.py`, `src/data/explore_ratings.py`) — ~1.07M
   films, of which ~13.8K have both budget and revenue.
3. Loaded MovieSum (`src/data/load_moviesum.py`,
   `src/data/explore_moviesum.py`): 2,200 screenplays across train / val /
   test (1,800 / 200 / 200), all with valid `tt`-prefixed IMDb IDs but
   only **2,188 unique** IDs — 12 same-IMDb-ID duplicate pairs of
   alternate-title or alternate-draft variants of the same film. XML
   structure verified to match `PROJECT_CONTEXT.txt` Section 4.
4. Joined the two datasets on IMDb ID directly
   (`src/data/join_corpus.py`). Both sources carry the IMDb ID natively,
   so the merge is a single line — no fuzzy matching, no external bridge.
   Deduped the ratings dataset on `imdb_id` first (a small fraction
   appears twice under different TMDB IDs — alternate cuts).
   **Result: 2,186 / 2,188 MovieSum films matched (99.9%); 1,713 of
   those also have non-zero budget AND revenue AND a rating.**
5. Profiled the working corpus (`src/data/profile_corpus.py`): year,
   genre, budget, revenue, rating, ROI, and screenplay length. Saved
   plots and a summary table.
6. Saved the joined dataset to
   `data/interim/phase1_joined_corpus.parquet` (one row per
   deduplicated MovieSum film, all 2,188 rows; the 1,713 four-signal
   subset is a filter on top, computed at load time in
   `src/data/profile_corpus.py`).
7. Wrote a dup-pair review helper (`src/data/review_duplicates.py`)
   that prints the 12 same-IMDb-ID pairs side-by-side and emits a
   fillable CSV at
   `reports/tables/phase1_moviesum_duplicates_review.csv` so the user
   can mark a `keep`/`drop` decision per pair. Phase 2 will encode the
   resulting policy.
8. Built `notebooks/phase_1.ipynb` — a strong-EDA narrative version of
   the phase, with embedded interpretations after every plot, ready to
   feed the final report.

## Why we did it that way

- **Direct IMDb-ID merge.** Both sources carry IMDb IDs natively, so
  there is no need for normalization, fuzzy matching, or external ID
  bridges. The merge is a single line of pandas; the precision is
  exact.
- **Dedup MovieSum by longest script.** The 12 same-IMDb-ID pairs differ
  either trivially (alternate spelling, same script) or substantively
  (alternate drafts). Keeping the longer script favors the more-complete
  draft, which matches what a downstream feature pipeline will treat as
  the canonical screenplay. The user is reviewing each pair manually
  via `review_duplicates.py` so Phase 2 can encode a documented policy.
- **Dedup the ratings dataset by `vote_count`.** Some films appear in
  the ratings dataset under more than one TMDB ID (alternate cuts,
  regional releases). Keeping the row with the higher `vote_count`
  picks the better-known variant, which is also the row most likely to
  carry budget/revenue/rating values.
- **`IMDB_Rating` preferred over TMDB `vote_average`.** The ratings
  dataset carries both. `IMDB_Rating` is better-populated on this
  matched subset and is the metric the report ultimately discusses.
  `vote_average` is kept as a fallback for the few rows where
  `IMDB_Rating` is missing.

## Tactical choices made

- **Logging via Python's `logging` module** (no `print` for diagnostic
  output) per `CLAUDE_CODE_GUIDELINES.txt` Section 3 — installed once
  via `src/utils/logging.py::get_logger`. Default level is WARNING; set
  `MSBA315_LOG_LEVEL=INFO` to see operational messages.
- **Logger output goes to stdout** (not stderr) so messages render as
  ordinary black-text output in Jupyter rather than red-stderr boxes.
- **`matplotlib.use("Agg")` is NOT called at module import time** —
  this lets notebooks pick whatever backend they like (e.g. inline). CLI
  scripts work without a display because we save with
  `fig.savefig` and never call `plt.show`.
- **Parquet output via `pyarrow`** for the working corpus; CSV for
  human-readable summary tables and review templates.
- **Random seed 42** in spot-check sampling, per project standard.
- **Brief-versus-actual file extension:** the foundation docs reference
  each other as `.md` but exist as `.txt` on disk (user preference,
  2026-05-02). The path resolver accepts both.

### Profile of the 1,713-film working corpus

- **Years:** 1932 – 2023, median 2005. Distribution dense ~1995–2022.
- **Budget:** median $25M; long right tail.
- **Revenue:** median $64M; very heavy right tail.
- **Rating** (`IMDB_Rating` preferred): mean ~7.1, narrow band — IMDb
  ratings tend higher than TMDB `vote_average` on the same films.
- **ROI (revenue/budget):** median ~2.9x; ~80% gross-profitable. Still
  survivor-biased (the corpus contains films that got produced and made
  it onto a major metadata aggregator), but less than would be the
  case with a smaller curated ratings source.
- **Screenplay length:** median ~211k characters (~34k tokens), narrow
  spread — consistent with feature-length conventions.

## Issues encountered & resolved

1. **MovieSum duplicate IMDb IDs (12 pairs).** Same film with alternate
   titles or alternate drafts. Resolved provisionally by keeping the
   longest script per IMDb ID; Phase 2 will encode the user's
   per-pair `keep`/`drop` decisions from `review_duplicates.py`.
2. **Ratings dataset has duplicate `imdb_id` rows.** Some films appear
   under multiple TMDB IDs (alternate cuts, regional releases).
   Resolved by deduping on `imdb_id` keeping the row with the higher
   `vote_count`.
3. **Future-year noise in `release_year_parsed`.** A handful of rows
   carry years like 2055 or 2099 (scheduled releases, error entries).
   Clipped to 1900–2025 at plot time; doesn't affect the join because
   the affected rows are obscure and rarely intersect with MovieSum.

## What we learned in EDA, and what Phase 2 will do about it

This is the consolidated checklist from the notebook's EDA. Each item
pairs an observation with a planned action. Phase 2 will not apply
items in the "open" section without checking with the user first; per
`PROJECT_CONTEXT.txt` Section 11, preprocessing decisions need
explicit confirmation before implementation.

### Already decided (Phase 2 encodes these directly)

1. **Pre-1995 tail of about 50 films.** Drop them. Corpus is dense
   1995-2022; the tail is too thin (under 5 films per year) to support
   era-stratified analyses. Decision logged 2026-05-02.
2. **Two MovieSum films absent from the ratings dataset.** Drop them.
3. **Ratings-dataset duplicates on `imdb_id`** (alternate cuts, regional
   releases). Dedupe by `imdb_id` keeping the row with the higher
   `vote_count`. Already implemented in `dedupe_ratings()`.
4. **Future-year noise (2055, 2099, etc.) in the ratings dataset.**
   Clip `release_year_parsed` to 1900-2025 when plotting or filtering;
   the affected rows rarely intersect MovieSum.
5. **Effective rating column.** Use `IMDB_Rating` when present and
   fall back to `vote_average`. Phase 2 standardizes on a single
   `effective_rating` column for downstream use.

### Decided in principle, Phase 2 must surface options before implementing

6. **Heavy right-skew in budget and revenue.** Plan: `log1p(budget)`
   and `log1p(revenue)` so outliers don't dominate. Phase 2 should
   present the user with options (plain `log1p`, `log10`, or quantile-
   bucketed) before applying.
7. **MovieSum dedup policy for the 12 same-IMDb-ID pairs.** User has
   completed the row-by-row review and filled in the `decision`
   column at `reports/tables/phase1_moviesum_duplicates_review.csv`.
   Phase 2 must read that file and encode the chosen policy
   (one `keep` row and one `drop` row per IMDb ID).
8. **Genre long tail** (Documentary, TV Movie, Foreign, Western, etc.
   with thin counts). For per-genre CV in Phase 4 we need to bucket
   them. Phase 2 (or 3) should ask: aggregate all into "other", drop
   them, or keep them with noisy per-genre estimates accepted.
9. **Pre-processing pipeline structure.** Imputation strategy, scaling,
   feature transforms. Phase 2 must present options and let the user
   pick (per `PROJECT_CONTEXT.txt` Section 11).

### Observations to carry forward into later phases

10. **Survivorship bias is substantial.** About 80% of the working
    corpus is gross-profitable, well above industry's roughly 50%
    net-profitable rate. Phase 6 cost matrix must use industry base
    rates, not corpus rates. May also want to reweight at training
    time in Phase 4; user should decide.
11. **Narrow rating range** (mean ~7.0, std ~0.7). Expect modest R^2
    in Phase 4. The asymmetric-cost decision layer in Phase 6 is where
    the system earns value, not the regression accuracy.
12. **No easy linear shortcut to outcomes.** Rating is only weakly
    correlated with budget, revenue, ROI, or screenplay length. The
    dialogue features extracted in Phase 3 must carry most of the
    predictive weight.
13. **Screenplay length vs. transformer windows.** Screenplays are
    about 34,000 tokens long on average. Most pre-trained text models
    used for feature extraction (BERT, DistilBERT, etc.) cap at 512
    tokens per pass. Phase 3 needs chunking (scene, paragraph, or
    fixed window) plus pooling, or a hierarchical approach. Strategy
    decision deferred to Phase 3.
14. **Per-decade thin cells.** Pre-1980s decades each have under 30
    films; the pre-1995 cutoff (item 1) handles most of this.
    Era-stratified CV in Phase 4 should bucket remaining decades.

## Open questions / things to flag

These need strategic input before Phase 2 starts:

1. **Survivorship bias framing.** ~80% of the working corpus is
   gross-profitable, well above industry base rate. Do we (a) keep the
   corpus as-is and frame the model as triaging *production-ready,
   recognizable* scripts (the realistic deployment target — studios
   mostly receive scripts from established writers anyway), or (b)
   reweight at training time toward a more representative
   profitability distribution? (a) is simpler and defensible; (b) is
   more ambitious and adds a methodological hook.
2. **Year scope.** The corpus is effectively 1995–2023. Decision
   already made (logged 2026-05-02): drop films released before 1995
   in Phase 2.
3. **Outcome variable.** Per `PROJECT_CONTEXT.txt` Section 8, Phase 4
   trains both rating and box-office models and compares. No action
   needed yet.
4. **MovieSum dedup policy.** User has filled in
   `phase1_moviesum_duplicates_review.csv` with `keep`/`drop` per row.
   Phase 2 reads the file and encodes the policy.

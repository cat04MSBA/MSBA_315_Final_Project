## Why we did it that way

- **MovieSum → TMDB direction (not TMDB → MovieSum).** MovieSum has IMDb
  IDs natively; TMDB 5000 does not. Building the lookup the other way
  would have required either the TMDB API (a previous attempt returned
  0/3,229 matches per the brief) or an external mapping. Title + year
  fuzzy is the brief's documented fallback, and it produces verifiable
  matches.
- **Title normalization (lowercase + strip punctuation + drop leading
  articles + lookup against `title` and `original_title`).** Conservative
  enough to avoid false positives (verified in spot-checks of 20 random
  matches — all correct) and broad enough to handle the punctuation
  differences between datasets (`X-Men: Days of Future Past` ↔
  `x men days of future past`).
- **Year tolerance ±1.** Festival- vs theatrical-release ambiguity and
  regional cuts can put the same film in adjacent years across datasets.
  18 of 1,121 matches (1.6%) used the ±1 offset; 0 used wider tolerance
  in the deployed strategy. Empirically, the join is dominated by exact
  year matches.
- **Dedup MovieSum by IMDb ID, keeping the longest script.** The 12
  same-ID pairs differ either trivially (alternate spelling, same script)
  or substantively (alternate drafts). Keeping the longer script favors
  the more-complete draft, which matches what a downstream feature
  pipeline will treat as the canonical screenplay.
- **No fuzzy library yet (no `rapidfuzz` install).** Verified empirically
  that the ceiling on the normalized-exact match is ~1,176 films and that
  obvious unmatched cases (e.g., "Ghostbusters_1984", "Bean_1997",
  "Frantic_1988", "Matchstick Men_2003") are absent from TMDB 5000
  entirely, not just absent under the current normalization. Adding fuzzy
  match would yield only marginal recovery — **the binding constraint is
  TMDB 5000's coverage, not the join algorithm**.

## Tactical choices made

- **Logging via `logging` module** (no `print`) per
  `CLAUDE_CODE_GUIDELINES.txt` Section 3 — installed once via
  `src/utils/logging.py::get_logger`.
- **Matplotlib `Agg` backend** for headless plotting from CLI scripts.
- **Parquet output via `pyarrow`** (already installed) for the working
  corpus; CSV for human-readable summary tables.
- **Random seed 42** in spot-check sampling, per project standard.
- **TMDB lookup keyed by both `title` and `original_title`** — widened
  recall without sacrificing precision (verified in spot-checks).
- **`vote_count` tiebreak** when a `(title, year)` resolved to multiple
  TMDB rows — picks the better-known film.
- **Brief-versus-actual file extension:** the foundation docs reference
  each other as `.md` but exist as `.txt` on disk; user confirmed
  2026-05-02 to keep new docs as `.txt`. The path resolver accepts both
  to stay robust to a later rename.

### Profile of the 1,011-film working corpus

- **Years:** 1934 – 2016, median 2003. Distribution dense ~1990–2015,
  long thin tail to 1934. **No 2017+ films** in the working set —
  TMDB 5000 effectively cuts off at 2016 (1 film in 2017, zero after).
- **Budget:** median $30M, mean $48M; long right tail.
- **Revenue:** median $90M, mean $174M; very heavy right tail.
- **Rating:** mean 6.69, median 6.70, narrow band (TMDB skews to
  recognized films which skew positive).
- **ROI (revenue/budget):** median 3.0x; **85.8% gross-profitable**
  (revenue > budget). This is far higher than industry base rate (~50%
  by net) and confirms strong survivorship bias in the corpus — the
  films are simultaneously *produced* (already filtered by greenlight)
  and *recognized enough to be in TMDB* (filtered by audience reach).
- **Screenplay length:** median 211k characters (~34k tokens), narrow
  spread — consistent with feature-length conventions.

### Saved figures and interpretations

- `phase1_tmdb_year_distribution.png` — TMDB films-with-both: dense
  1997–2016 (~100–250/yr), long thin tail to 1916. Confirms the prior
  documentation. **Implication:** the join's recall ceiling for any
  post-2016 MovieSum film is essentially zero against TMDB 5000.
- `phase1_tmdb_genre_distribution.png` — Drama, Comedy, Thriller,
  Action lead by a wide margin; documentaries, foreign, and small
  genres are sparse. **Implication:** any per-genre breakdown later
  in the project should bucket the long tail or use stratified
  sampling rather than relying on raw counts.
- `phase1_moviesum_length_distribution.png` — bell shape centered
  ~210k chars with light heavy-tail. **Implication:** screenplays are
  long-form (mean ~34k tokens) — Phase 3 feature extraction needs to
  handle inputs that exceed any single transformer context window.
  Chunking or scene-level pooling will be required.
- `phase1_joined_year_distribution.png` — joined-corpus year
  distribution dense 1995–2016, no 2017+ entries. **Implication:** the
  corpus reflects roughly two decades of mainstream cinema; era
  breakdowns in Phase 4 should respect this skew.
- `phase1_joined_genre_distribution.png` — same Drama/Comedy/Action
  lead. **Implication:** balanced enough to do per-genre
  cross-validation slices in Phase 4 without thin cells, except for
  documentaries and rare genres which should be aggregated.
- `phase1_joined_budget_revenue_distribution.png` — both budget and
  revenue heavily right-skewed; log-transformed scales are required
  for downstream modelling and visualization. **Implication:** Phase 3
  will use `log1p(budget)` and `log1p(revenue)` (or quantile-binned
  tier targets) rather than raw dollar values.
- `phase1_joined_rating_distribution.png` — narrow Gaussian-ish
  centered ~6.7. **Implication:** the regression target has limited
  dynamic range relative to noise; we should not expect huge
  predictive R² and the asymmetric-cost decision layer (Layer 3)
  becomes the practically meaningful contribution.
- `phase1_joined_roi_distribution.png` — log10(ROI) centered roughly
  at 0.5 (3x), heavy upper tail to 10x+, thin left tail of flops.
  **Implication:** the corpus over-represents profitable films;
  Phase 6 cost matrices must be calibrated to industry base rates,
  not corpus base rates, or sensitivity-analyzed across both.
- `phase1_joined_screenplay_length.png` — same bell shape as the full
  MovieSum length plot, similar mean. **Implication:** the four-signal
  filter doesn't preferentially keep short or long screenplays.

  Strategic input is needed before Phase 2 — see "Questions for the

  biased toward films that succeeded after greenlight. The
  pre-greenlight framing must keep acknowledging this throughout the
  report; Phase 6's cost matrix should be tied to industry base rates,
  not corpus rates.

  1995–2016. Era-based generalization claims to films before 1980 or
  after 2016 will be weakly supported.

  transforms before modeling; raw scale is unusable.

## Questions for the planning conversation

1. **Corpus size decision.** The working corpus is 1,011 films, which
   per `PROJECT_CONTEXT.txt` Section 5 falls in the "1,000–1,500:
   proceed with MovieSum primary, document as a limitation" band.
   Which path does Phase 2 take?

   - **(A) Accept 1,011 as the corpus, document the limitation.**
     Phase 2 builds the pipeline on this set as-is. Lowest effort;
     strongest survivorship-bias caveat in the report.
   - **(B) Replace TMDB 5000 with a larger TMDB dump.** Kaggle and
     other sources host fuller TMDB exports (10k+, 100k+, ~all-of-
     TMDB). Could plausibly recover the 1,012 MovieSum films currently
     missing because TMDB 5000 doesn't cover them. Highest expected
     gain; main risk is data-quality variance in larger dumps and one
     additional download/cleanup pass.
   - **(C) Add IMSDb supplementary scripts (the brief's documented
     fallback).** Adds screenplays the original plan envisioned. Lower
     script quality (raw scrapes) and would still need the same TMDB
     join, so not obvious that it lifts the four-signal count above
     ~1,200–1,300.
   - **(D) Drop the budget+revenue requirement for some films and
     train two outcome variables independently.** Already aligned with
     the deferred-model decision in `PROJECT_CONTEXT.txt` Section 8;
     would let us train a rating-prediction model on ~1,121 films
     (matched, with rating) and a box-office model on the smaller
     1,011 subset. Phase 4 would still benchmark both as planned.

   The recommendation, if forced to pick: **(B) primary, (D) as a
   complement.** A larger TMDB export is a one-time data-prep cost that
   pays off across all later phases; (D) keeps Phase 4 honest by
   acknowledging that rating-prediction and box-office-prediction don't
   need the same corpus.

2. **Survivorship bias framing.** Given the 85.8% profitable rate, do
   we (a) keep the corpus as-is and emphasize that the model predicts
   "well-known production" outcomes (i.e., the realistic deployment
   target — studios mostly receive scripts from established writers
   anyway), or (b) explicitly downweight the popularity skew during
   training (e.g., propensity-style reweighting toward an industry-
   base-rate distribution)? (a) is simpler and defensible; (b) is more
   ambitious and adds a methodological hook.

3. **Year scope.** The corpus is effectively 1995–2016. Should Phase 2
   restrict to 1995+ explicitly (drops the long thin tail to 1934 — ~30
   films) so that downstream temporal generalization claims are clean,
   or keep the older films and stratify CV by era?

4. **MovieSum dedup policy.** Phase 1 keeps the longest script per
   IMDb ID. Phase 2 should formalize this — alternative policies:
   "keep MovieSum's `train` split if available, else `val`, else
   `test`" (keeps the dataset's intended splits intact for any
   reproducibility cross-check), or a manual review of the 12 pairs.
   Policy preference?

5. **TMDB API or mapping dataset for direct IMDb ID joins?** Even if
   we keep TMDB 5000, having an IMDb→TMDB ID mapping would let Phase 2
   use exact joins instead of fuzzy ones. There are static mapping
   files on Kaggle and the OMDb API offers free tier ID lookups. Worth
   spending an hour on for cleaner downstream data, or unnecessary
   given the 95% precision we already see?

# Phase 1 Brief — Data Feasibility Verification

> **Read first:** `docs/PROJECT_CONTEXT.md` (project framing, decisions,
> mandatory checkpoints) and `docs/CLAUDE_CODE_GUIDELINES.md` (engineering
> standards, escalation rules, summary format).
>
> This is the first execution brief. It is intentionally exploratory rather
> than production-quality. Phase 2 will rebuild the data pipeline cleanly;
> Phase 1 just answers the question: do we have enough usable data to
> proceed at the planned scale?


---

## 1. Goal of This Phase

Answer one question: **how many films will we end up with that have all
four required data points (screenplay, IMDb rating, budget, revenue)?**

The answer determines whether the project proceeds with MovieSum-only
(target ≥1,500 films), MovieSum + IMSDb supplementary (target ~1,500
after dedup), or scope reduction (<1,000 films).

Decision criteria are documented in `PROJECT_CONTEXT.txt` Section 5.

---

## 2. Inputs You Will Receive

The user has already downloaded both raw datasets and will drop them into
`data/raw/` after you create the folder structure (Task 1).

Expected files:

- **IMDb-TMDB Movie Metadata Big Dataset (1M)** — from Kaggle. The relevant
  file is `IMDB TMDB Movie Metadata Big Dataset (1M).csv`, ~950 MB,
  ~1.07M rows. Carries `imdb_id` (`tt`-prefixed) and TMDB `id` natively,
  plus budget, revenue, runtime, genre list, release date, multiple rating
  fields (TMDB `vote_average`, IMDb `IMDB_Rating`, Metacritic `Meta_score`),
  director, production companies, and other metadata. Stored under
  `data/raw/ratings_data/`.
- **MovieSum** — from `https://huggingface.co/datasets/rohitsaxena/MovieSum`.
  2,200 movie screenplays in structured XML format, each tagged with an
  IMDb ID. Stored under `data/raw/script_data/` as three JSONL split files
  (`train.jsonl`, `val.jsonl`, `test.jsonl`). Format details documented in
  `PROJECT_CONTEXT.txt` Section 4.

**Workflow:** After completing Task 1 (folder setup), pause and ask the
user to confirm the data files are in place at `data/raw/`. Once
confirmed, continue with the remaining tasks.

---

## 3. What Has Already Been Done (Sanity Check)

The ratings dataset is the user's curated download; we have no prior
exploration numbers to reproduce. Use the `summarize_ratings` helper in
the loader to get headline counts (total / with-budget / with-revenue /
with-both / with-imdb_id) and sanity-check that:

- The total is in the order of ~1M rows (the file is ~950 MB).
- A non-trivial fraction has both budget AND revenue (expected: ~13K).
- Most rows have a valid `imdb_id` (expected: ~590K).

Both datasets carry IMDb IDs natively, so the MovieSum→ratings join is a
direct exact-ID merge — no fuzzy matching, no external bridge.

---

## 4. Tasks

This phase is exploratory. Focus on understanding the data and answering
the corpus-size question. Production-quality code lives in Phase 2; here
the goal is to see clearly.

### Task 1: Set up project structure

Create the folder structure documented in `PROJECT_CONTEXT.md` Section 7,
to the extent needed for Phase 1:

- `src/data/` — for any data-loading scripts
- `src/utils/` — for logging and paths utilities
- `data/raw/` — where the user's downloaded files will live
- `data/interim/` — for any intermediate Parquet files saved this phase
- `reports/figures/` — for diagnostic plots
- `docs/summaries/` — for the phase summary you'll write at the end

Add a `.gitignore` that excludes `data/raw/`, `data/interim/`,
`data/processed/`, and standard Python artifacts.

Set up a minimal `src/utils/paths.py` that resolves the project root
(walk up from the file location until you find a marker, e.g.,
`docs/PROJECT_CONTEXT.md`) and exposes paths to the data directories.
Set up `src/utils/logging.py` to configure a project logger at INFO level.

**At the end of Task 1, pause and tell the user:**

> "Folder structure is ready. Please drop `tmdb_5000_movies.csv` and the
> MovieSum files into `data/raw/`, then confirm so I can proceed."

Do not begin Task 2 until the user confirms.

### Task 2: Load the ratings dataset and profile it

Write `src/data/load_ratings.py` with a function that loads
`IMDB TMDB Movie Metadata Big Dataset (1M).csv` and returns a clean
DataFrame. Read column-selectively — the file is ~950 MB and we only
need ~20 of the 42 columns for downstream use.

In the loader or in a separate exploration script:
- Headline counts via `summarize_ratings` (total / with-budget /
  with-revenue / with-both / with-imdb_id)
- Print and save the year distribution of films-with-both
- Print and save the genre distribution of films-with-both
- Compute and save a few sanity statistics (budget median/mean/std,
  revenue median/mean/std, revenue/budget ratio distribution)

### Task 3: Load MovieSum and explore its structure

Write `src/data/load_moviesum.py` with a function that loads MovieSum.

Inspect the structure:
- Total number of screenplays
- IMDb ID coverage (how many entries have a valid `tt`-prefixed IMDb ID)
- Year distribution if year information is available
- A sample screenplay's XML structure: parse one to confirm the format
  matches what's documented in `PROJECT_CONTEXT.md` Section 4
- Length distribution (token counts or character counts per screenplay)

Spot-check 5 random screenplays by hand — confirm they look like real
screenplays with scenes, characters, dialogue, and stage directions. If
the structure differs from what's documented, flag it.

### Task 4: Join MovieSum and the ratings dataset on IMDb ID

This is the central question of Phase 1.

- Both datasets carry `imdb_id` natively, so the join is a direct exact-ID
  merge in `src/data/join_corpus.py`.
- Deduplicate the ratings dataset on `imdb_id` first (a small fraction of
  films appear twice under different TMDB IDs — alternate cuts, regional
  releases). Keep the row with higher `vote_count`.
- After merging, count: matched, matched + budget>0, matched + revenue>0,
  matched + both, matched + all four (with rating). The last is the
  working corpus size.
- Spot-check 20 random matches by eye to verify the join is sane.

### Task 5: Profile the joined corpus

For the joined films with all four data points (screenplay + rating +
budget + revenue), produce diagnostic plots and statistics:

- Year distribution
- Genre distribution
- Budget distribution (raw and log-scale)
- Revenue distribution (raw and log-scale)
- IMDb rating distribution
- ROI distribution (revenue / budget)
- Screenplay length distribution

Save plots to `reports/figures/` with descriptive filenames
(`phase1_year_distribution.png`, etc.).

For each plot, write a one-to-two-sentence interpretation in the phase
summary — what the plot shows and what it implies for the project.

### Task 6: Save intermediate artifacts

Save the joined working dataset (one row per film, all available columns)
as Parquet to `data/interim/phase1_joined_corpus.parquet`. Phase 2 will
rebuild this properly; the Phase 1 version is for inspection and to
support the recommendation in the phase summary.

### Task 7: Write the phase summary

Use the template in `docs/CLAUDE_CODE_GUIDELINES.md` Section 7. Save to
`docs/summaries/phase_1_summary.md`.

Update `docs/PROJECT_CONTEXT.md` Section 5 (Data Summary) and Section 9
(Phase Status) with the actual numbers found.

This phase ends with a mandatory checkpoint
(`PROJECT_CONTEXT.md` Section 11). At completion, prepare the following

> "Phase 1 complete. Please bring the summary and these questions to the

Specific questions to surface:

2. What is the year/genre/budget distribution? Any concerning skews?
3. Was the join method clean (direct ID match) or fuzzy (title+year)? If

5. Anything unexpected in the data that affects the project framing or

---

## 6. Anti-Goals for This Phase

Things to NOT do in Phase 1:

- Do not build production-quality data pipeline code. That is Phase 2's
  job. Phase 1 code can be exploratory and a bit rough.
- Do not extract any features from screenplays. That is Phase 3.
- Do not start any modeling or train/test splitting. That is Phase 3
  onward.
- Do not over-clean the data. The goal is to understand it, not perfect it.
- Do not attempt to fix MovieSum or TMDB if data quality issues are found.
  Document the issues and surface them in the phase summary.

---

## 7. When to Ask vs. When to Act

Following `CLAUDE_CODE_GUIDELINES.md` Section 6:

**Act on your own** for tactical choices like which fuzzy-match library
to use, what threshold to set, plot styling, file naming, etc. Briefly
state the rationale in the phase summary.

**Ask the user in chat** if:
- The join strategies above all fail and you need direction
- The data files are corrupt, missing, or in a different format than
  documented
- You hit an environment issue (missing library, version conflict)
- Anything in this brief is ambiguous

**Tell the user to escalate to the planning conversation** if:
- The corpus size after joining is unexpectedly far below 1,000 films
- You discover a fundamental issue with one of the data sources that
  affects the project framing
- You believe a different data strategy would serve the project better
  (per `PROJECT_CONTEXT.md` Section 11 "Proposing Alternatives")

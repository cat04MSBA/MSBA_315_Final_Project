# `data/` — How to populate this folder

The contents of `data/raw/`, `data/interim/`, and `data/processed/` are
**not committed** to git (the raw files alone are ~1.4 GB, well over GitHub's
limits). Only the folder structure is tracked, via `.gitkeep` placeholders.

To rebuild the working dataset from scratch, download the two raw sources
into `data/raw/` and run the build pipeline. Everything in `data/interim/`
and `data/processed/` is regenerated by code.

---

## 1. Expected layout

```
data/
├── raw/
│   ├── script_data/
│   │   ├── train.jsonl                 # MovieSum
│   │   ├── val.jsonl                   # MovieSum
│   │   └── test.jsonl                  # MovieSum
│   └── ratings_data/
│       └── IMDB TMDB Movie Metadata Big Dataset (1M).csv
├── interim/                            # built by pipeline (Phase 1 join)
└── processed/                          # built by pipeline (Phase 2 corpus)
```

---

## 2. Where to download the raw data

### MovieSum (screenplays) — `data/raw/script_data/`

- Source: Hugging Face dataset [`rohitsaxena/MovieSum`](https://huggingface.co/datasets/rohitsaxena/MovieSum)
- Reference: Saxena & Keller, ACL 2024
- ~50–100 MB total across `train.jsonl`, `val.jsonl`, `test.jsonl`
- License: research use

### IMDb-TMDB Movie Metadata (ratings) — `data/raw/ratings_data/`

- Source: Kaggle — *IMDB TMDB Movie Metadata Big Dataset (1M)*
- File needed (exact name, including spaces and parentheses):
  `IMDB TMDB Movie Metadata Big Dataset (1M).csv`
- ~1 GB
- License: see Kaggle dataset page (research / non-commercial use)

---

## 3. Building `interim/` and `processed/`

Once both raw sources are in place, run the pipeline from the project root:

```bash
python3 -m src.data.build_corpus              # builds data/processed/
python3 -m src.data.validate_processed_corpus # sanity-checks the build
```

`build_corpus` is idempotent and produces:

- `data/processed/films_joined.parquet` — master per-film table (1,713 × 41)
- `data/processed/screenplays_parsed.pkl` — parsed scene/dialogue structure

See `docs/DATA_NOTES.md` for the full column glossary and known caveats
(survivorship bias, year-density skew, etc.).

---

## 4. Regenerating Phase 3 artifacts on a fresh checkout

Several Phase 3 artifacts under `data/processed/` are gitignored and
must be regenerated after a fresh clone. From the project root, once
`films_joined.parquet` and `screenplays_parsed.pkl` exist:

```bash
# Train / calibration / test split (70/15/15, seed 42, stratified by
# (primary_genre_bucketed, decade_bucket)). Produces
# data/processed/split_assignments.parquet plus a diagnostic table at
# reports/tables/phase3_split_diagnostics.csv. Idempotent: same seed
# produces byte-identical output across runs.
python3 -m src.features.split

# Lexical feature matrix (14 features per film). Produces
# data/processed/features_lexical.parquet. Requires nltk + wordfreq
# (installed via requirements.txt). Takes ~10 minutes on a single CPU.
python3 -m src.experiments.run_lexical_ablation

# Sentiment feature matrix (22 features per film). Produces
# data/processed/features_sentiment.parquet. Requires nltk + nrclex.
# Takes ~2 minutes on a single CPU.
python3 -m src.experiments.run_sentiment_ablation
```

Each of these is an experiment runner: in addition to the parquet
feature matrix, it produces a self-contained directory under
`runs/phase_3/<YYYYMMDD_HHMM>_<name>/` and appends one row to
`reports/tables/phase3_ablation.csv`. The Phase 3 multi-family baseline
table at `reports/tables/phase3a_baseline.csv` is regenerated by
`python3 -m src.models.baseline.train` and depends on
`split_assignments.parquet`.

NLTK resources (`punkt`, `punkt_tab`, `vader_lexicon`) are downloaded
on demand at module-load time by the feature modules'
`ensure_nltk_resources` helpers; no manual `nltk.download` step is
required.
